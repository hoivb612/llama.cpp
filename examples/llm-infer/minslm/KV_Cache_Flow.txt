  KV Cache Flow

  1. Initialization (llm_initialize, lines 139-297)

  Two modes:

   - Non-PFC mode: Simple — KV cache starts empty, nothing cached.
   - PFC (Prefix Cache) mode (lines 240-294):
    - Parses custom_template_prompt looking for {message} placeholder
    - The portion before {message} becomes pfx_shared (the shared prefix)
    - Tokenizes this prefix into llm_tokens_shared
    - Attempts to load a saved state file via llama_state_load_file() — this restores the full KV cache from a
  previous session
    - If load fails, marks save_llm_state = true to save on first inference

  2. Inference (llm_inference, lines 299-510)

  The KV cache is populated in two phases:

  Phase 1 — Prompt processing / prefill (lines 306-412):

   - PFC mode (lines 310-321): Removes "future" tokens from KV cache with llama_kv_self_seq_rm(), sets n_past to the
  shared prefix length. Only new tokens (after the shared prefix) need decoding.
   - Non-PFC mode (lines 323-329): Clears KV cache entirely with llama_kv_self_clear(), starts from scratch.
   - Lines 354-371: Calculates prompt_index — how many tokens from the input match the cached prefix. Only tokens
  after this index need to be decoded.
   - Lines 399-412: Actual KV cache generation — calls llama_decode() in batches of n_batch size. Each llama_decode()
  call processes tokens through all transformer layers and populates the KV cache for those positions.

  Phase 2 — Token generation / autoregressive (lines 457-492):

   - Samples one token at a time via llama_sampler_sample()
   - Each new token is decoded with llama_decode(llm_ctx, llama_batch_get_one(&embd[0], 1)) — this adds one KV entry
  per call
   - n_past increments by 1 each iteration

  3. State saving (lines 431-446):

   - On first inference in PFC mode, saves the full state (model weights + KV cache) via llama_state_save_file() so
  subsequent runs can skip prefill.
