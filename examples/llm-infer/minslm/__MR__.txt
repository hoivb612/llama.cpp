
Layer Windowing System — Phases 1-4

Problem

GGUF models load ALL transformer layers into memory at startup, even on memory-constrained systems. A 1.4 GB model occupies 1.4 GB
regardless of available RAM.

Solution

A layer windowing system that only keeps N layers resident at a time, streaming the rest from NVMe/mmap on demand during compute.

---------------------------------------------------------------------------------------------------------------------------------

Phase 1: Instrumentation

 - Added per-layer weight size reporting in load_all_data() (gated by GGML_MODEL_LAYERS_STAT=1 env var)
 - Shows non-layer breakdown (token_embd, output, output_norm), per-layer min/max/avg, and a budget table estimating how many 
layers fit at 64/128/256/512/1024/2048 MiB

Phase 2: Infrastructure

 - Created llama-layer-window.h/.cpp — the layer_window_manager struct tracking per-layer size, residency, and tensor locations
 - Added --weight-budget MB CLI parameter to minslm tools, which sets GGML_WEIGHT_BUDGET_MB env var
 - Modified load_all_data() with a two-pass approach: pre-scan all tensors for layer sizes, then skip loading deferred layers 
(layers that don't fit within the budget)

Phase 3: Deferred Loading with On-Demand Reload

 - For deferred layers in mmap mode: still sets up the tensor data pointer (so the graph can reference it) but skips mlock — pages
 aren't pinned in physical RAM
 - Added ensure_layer_resident() — calls VirtualLock (Windows) or mlock (Linux) to pin mmap pages when a layer is needed
 - Added record_tensor_location() — stores file offset, size, and tensor pointer for each deferred tensor so it can be reloaded 
later
 - Stored mmap base addresses from load_tensors() before mappings are moved
 - Initially hooked as "load all deferred layers at first compute" — proved the mechanism works (14 layers loaded in 60ms =
  7.4 GB/s)

Phase 4: True Per-Layer Windowing (Eval Callback)

 - Leverages ggml_backend_sched_eval_callback — fires per graph node during compute
 - layer_window_eval_callback() detects layer boundaries by inspecting tensor source names for blk.N. patterns
 - At each layer transition: calls ensure_layer_resident(new_layer) then evict_to_budget(new_layer)
 - Eviction: evict_layer() uses VirtualUnlock + DiscardVirtualMemory (Windows) or munlock + madvise(DONTNEED) (Linux) to release 
physical pages back to the OS while keeping the virtual mapping valid
 - LRU strategy: evict_to_budget() finds the oldest-accessed layer (excluding the current layer and initially-loaded layers 
without recorded tensor locations) and evicts until resident_bytes ≤ budget_bytes
 - Per-pass statistics: begin_pass()/end_pass() track and print loads/evicts per forward pass
 - Replaces the Phase 3 hook — the eval callback is set before graph_compute() and restored after

Additional: Vulkan UMA

 - Enabled buffer_from_host_ptr = true for integrated GPUs in the Vulkan backend, so mmap-based windowing works on Vulkan UMA 
systems the same as DX12 UMA

Key Design Decisions

 - Budget applies to layers only — non-layer tensors (embeddings, output head) are always resident as mandatory overhead
 - mmap-based eviction — no data copying; just pin/unpin physical pages via OS APIs
 - Initially-loaded layers are never evicted — they have no recorded tensor locations (loaded normally), so they form the 
permanent "base window"
 - Eval callback minimizes sync points — returns true (observe) only at layer boundaries, letting nodes within a layer batch 
together

 1. VRAM not reduced:

  This is the key limitation of the current design. The buffer allocation happens in load_tensors() before load_all_data():

   load_tensors() → ggml_backend_alloc_ctx_tensors_from_buft()  // allocates ALL tensors
   load_all_data() → fills data (with windowing skip)            // controls which get loaded

  The GPU buffer for all tensors is pre-allocated upfront — windowing only controls which pages are faulted into physical RAM. On
  your system with DX12 mmap (buffer_from_host_ptr), the GPU maps the entire mmap region. VirtualLock/DiscardVirtualMemory controls
  physical page residency, but the virtual address space and reported VRAM size stay the same.

  True VRAM reduction would require deferring buffer allocation for non-resident layers (Phase 5) — a much bigger change to the
  tensor allocation pipeline.

  What windowing DOES save today: physical RAM working set. The OS can reclaim those pages for other processes. Task Manager's
  "working set" should reflect the reduction even though "committed" memory stays the same.

  2. Backend compatibility:

  Yes — the windowing logic lives in load_all_data() which is backend-agnostic. It works for:

   - CPU ✅ — weights in host memory, mmap page control works directly
   - DX12 ✅ — buffer_from_host_ptr maps host mmap, page faults go through OS
   - Vulkan — depends on whether it uses mmap or copies to device-local VRAM. If it copies, the skip logic in load_all_data prevents 
  the copy for deferred layers, but the buffer is still allocated
   - CUDA — similar to Vulkan

  The VirtualLock/madvise eviction code is host-memory specific, so it works best with mmap-backed backends (CPU, DX12 on your
  system).

What happened: 

 - Model loads with only 21/35 layers from disk (fast startup)
 - On first inference call, the 14 deferred layers are loaded from mmap at
  7.4 GB/s (60ms!)
 - After that, all layers are resident and inference runs at full speed
 - Zero quality impact — same tokens per second, same output

The 60ms reload penalty is essentially free for interactive use. And on a UMA system where mmap maps directly to GPU-accessible
memory, this would be even faster (just page faults, no copy).

What we have now (Phase 3 — "Deferred Loading"):

 - Model loads fast: only 21/35 layers read from disk at startup
 - BUT: GPU buffer space for ALL 35 layers (1144 MiB) is still pre-allocated
 - On first inference: the 14 skipped layers are loaded from mmap → all 35 now resident
 - From that point on: everything is in memory — identical to no windowing
 - Benefit: faster app startup. No runtime memory savings.

True per-layer windowing (what I called Phase 4):

 - Only allocate buffer space for N layers at a time (e.g., N=8 at ~270 MiB)
 - During each forward pass (every token):
  - Load layer 0 weights into window slot → compute → evict
  - Load layer 1 weights into window slot → compute → evict
  - ... cycle through all 35 layers
 - At no point are all 35 layers in memory simultaneously
 - Benefit: actual runtime memory reduction — from 1144 MiB to ~270 MiB of weight memory

Why Phase 4 is hard: The compute graph is monolithic — all 35 layers are baked into one graph. We can't pause mid-graph to swap
weights. We'd need to either:

 1. Break the graph into per-layer sub-graphs (major architecture change)
 2. Or intercept the scheduler between layer dispatches to swap buffer contents

The honest summary: What we have now proves the infrastructure (tracking, deferred I/O, mmap integration) but the real memory win
requires restructuring how the graph executes. That's a much deeper change.

Actually, let me step back and think about what's practical and impactful right now. On this mmap system:

 - All weight data pointers already point into the mmap
 - The GPU reads from these host pointers (buffer_from_host_ptr on UMA)
 - The OS page cache already manages what's physically resident
 - We can use madvise/VirtualUnlock to hint the OS

The simplest Phase 4: rather than the eval callback (which adds per-node overhead), do a per-forward-pass windowing with madvise
hints:

 1. Before graph compute: madvise(WILLNEED) for all layers (prefetch)
 2. After graph compute: madvise(DONTNEED) for layers outside the window

But this doesn't actually limit peak memory — the GPU reads all layers during the forward pass.

For TRUE memory limiting, we need per-layer control during compute. The eval callback is the right tool.
